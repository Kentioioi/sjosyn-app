## 1. [HIGH] Alarm-mode burst sleeps (await sleep 5s ×3) inside the detection loop massively widen the overlap/timeout window for the ENTIRE poll
- **Dim:** backend-detect  **File:** netlify/functions/bg-poll-trigger.mjs 220-241
- **Scenario:** A subscriber with alarmMode='alarm' has a crossing. The send block runs burst=3 with `await sleep(ALARM_BURST_GAP_MS=5000)` between sends (lines 223-224), so this one firing blocks the loop ~10s. Multiple alarm-mode firings in one tick stack additively. Meanwhile all OTHER subscribers/tripwires after this record in the loop wait. If total tick time exceeds the cron interval or Netlify's function timeout (default 10s for sync functions; 26s hard cap), the function is killed mid-loop before line 252 writes state — OR the next cron tick overlaps (see lost-update finding).
- **Missed how:** If the function times out before reaching `stateStore.setJSON('all', stateAll)` at line 252, NONE of this tick's lastPositions/lastFired updates persist. Every crossing detected this tick after the slow alarm record — and the baseline advance for every vessel processed — is lost. Vessels that crossed are checked against stale prev next tick and their straddling segment is gone. Synchronous (non-background) Netlify functions have a 10s default execution limit; 3×5s bursts alone blow past it.
- **Fix:** Never block the detection loop on wall-clock sleeps. Send all bursts non-blockingly (collect send promises, await Promise.allSettled at the end) or, for spaced repeats, schedule them via a separate background function / delayed queue. Persist state EARLY (write lastPositions before doing any sends) so a timeout during sends cannot lose the baseline. Confirm this is deployed as a background function (`-background` suffix or config) given the burst design.

## 5. [MEDIUM] Mid-burst send error (non-404/410) breaks the whole record's tripwire loop — every LATER tripwire that ticked is silently skipped
- **Dim:** backend-detect  **File:** netlify/functions/bg-poll-trigger.mjs 237-242
- **Scenario:** A push subscriber has 2+ armed tripwires that all need checking this tick (e.g. tripwire A for vessel X and tripwire B for vessel Y both cross). The loop at line 183 processes tripwire A first; vessel X crossed, so it enters the send block. webpush.sendNotification throws a transient error with a statusCode that is NOT 404/410 (e.g. 429 Too Many Requests from the push service, 500/502/503 from FCM/Mozilla autopush/Apple, or a TLS/network timeout with no statusCode at all). - **Missed how:** `if (aborted) break` at line 242 terminates the entire per-record tripwire loop on any transient send failure of an EARLIER tripwire. The crossing is permanently lost, caused by an unrelated wire's transient push error.

## 19. [BY-DESIGN] Vessel crosses and returns within one poll interval → prev/cur never straddle the line (sub-poll-period transit)
- **Dim:** backend-detect  **File:** netlify/functions/bg-poll-trigger.mjs 199-205
- **Scenario:** Cron interval is 3 min (CLAUDE.md) / 1 min (code comment). Vessel X is at position P_t0 (one side of the line) at tick T. Between tick T and tick T+1 (1–3 min later) it crosses the tripwire line and crosses back, returning to the same side near P_t1. At tick T+1, prev=P_t0 and cur=P_t1 are both on the same side; segmentsIntersect(prev,cur,a,b) may be false (the straight chord doesn't cross), or for a corridor insideCorridor(prev) and insideCorridor(cur) are both the same so triggered=false.
- **Missed how:** Transition detection only samples positions once per poll interval. Any crossing event that begins and ends between two samples is invisible: the prev→cur chord never straddles the segment, so `triggered` is false and no push is sent. A fast vessel near a short tripwire can fully transit a corridor in under 3 minutes.
- **Fix:** Fundamental sampling limitation of minute-scale polling; reduce interval (cost tradeoff already documented) or move to the AIS SSE stream for continuous detection. For corridors, latch an 'inside' flag in state and fire on the next outside sample even if intermediate samples were missed, which catches more in/out cases than the single-segment test.
