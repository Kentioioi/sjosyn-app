# Tripwire review — confirmed findings (29)

_Generated by adversarial review workflow wf_42b86898-551. 4 findings rejected._

## 1. [HIGH] SW focus detection can route a crossing to an in-app banner that never shows, dropping the notification entirely
- **Dimension:** correctness
- **File:** src/sw.js 47-64; App.jsx:196-217
- **Problem:** On push the SW picks focused = a client that is focused OR visibilityState==='visible' and, if found, posts tripwire-event and RETURNS without ever calling showNotification. The in-app banner only renders if App's serviceWorker 'message' listener is mounted and the React tree is alive. There are real states where a client is reported visible/uncontrolled but cannot show the banner: (a) the page is mid-navigation or the React app crashed/not yet hydrated; (b) includeUncontrolled:true returns a client not controlled by this SW (e.g. a duplicate tab on an old build whose message handler shape differs); (c) the document is visible but the OS notification would still be the only thing the locked-attention user sees. In all these the OS notification is suppressed and the postMessage lands nowhere, so the user gets NOTHING for a one-shot crossing — and the wire is still deleted on the backend.
- **Consequence:** For a one-shot crossing, the user can receive no alert at all while the backend consumes (deletes) the tripwire.
- **Fix:** Treat the in-app banner as best-effort: post the message AND set a short timeout / use a client ack (MessageChannel) — if no client acks within ~500ms, fall back to self.registration.showNotification. Also require c.focused specifically (not merely visible) before suppressing the OS notification.

## 2. [HIGH] Frontend full-list re-sync resurrects a fired one-shot tripwire (sync vs SW postMessage delete race)
- **Dimension:** races-state
- **File:** src/utils/pushSubscribe.js / src/App.jsx / src/sw.js pushSubscribe.js:93-109; App.jsx:200-203,367-370; sw.js:40-64
- **Problem:** One-shot removal must stay consistent across 3 places: backend rec rewrite (bg-poll line 203-204), frontend localStorage/state (App handleRemoveTripwire), and the SW 'tripwire-fired' message. The intended flow: backend fires → rewrites rec WITHOUT the tripwire → pushes → SW receives push → posts 'tripwire-fired' → App removes it locally → the tripwires-change effect re-syncs the now-shorter list. But syncTripwiresToBackend sends the ENTIRE current tripwires map every time (filter+map over all entries, line 96-98) and push-subscribe OVERWRITES rec.tripwires wholesale (push-subscribe.mjs line 74-82). The SW only posts 'tripwire-fired' when the device receives the push. Push delivery is async and may be delayed/dropped, the app may be closed, or — if the app is FOCUSED when an UNRELATED tripwire change happens — App's effect (App.jsx:367) re-syncs the full local list (which still contains the fired tripwire because the 'tripwire-fired' message hasn't arrived yet) and overwrites the backend's already-pruned rec, putting the fired tripwire back.
- **Consequence:** Backend deletes one-shot tripwire from rec; a frontend sync that races ahead of (or never receives) the 'tripwire-fired' message re-uploads the full list including that tripwire → it is armed again on the server → it fires a SECOND time on the next crossing (or even the same crossing if lastFired in stateAll was also lost per other findings). The 'engangs' (one-shot) contract is violated whenever push delivery is slower than the next frontend sync, which is common.
- **Fix:** Make removal authoritative server-side and idempotent against re-sync: have push-subscribe MERGE rather than blindly replace (e.g. server keeps a per-endpoint set of 'firedOnceMmsi' and strips any incoming tripwire whose mmsi is in that set until the client explicitly acknowledges). Or include a per-tripwire monotonic version/firedAt from the server in the sync round-trip so the client cannot re-arm a tripwire the server marked fired. Do not treat the client's full-list PUT as ground truth for one-shot state.

## 3. [HIGH] 'tripwire-fired' postMessage can be lost → fired tripwire never removed from localStorage, re-synced forever
- **Dimension:** races-state
- **File:** src/sw.js / src/App.jsx sw.js:51-53; App.jsx:196-217
- **Problem:** The ONLY mechanism that removes a fired one-shot tripwire from frontend localStorage is the SW 'tripwire-fired' postMessage (App.jsx:200-203). The SW posts it to clients only inside the push event handler (sw.js:52-53), i.e. only if/when a push is actually delivered to THIS device AND a client window exists to receive it. If the app is fully closed (no client, push shown as OS notification only), `clients.forEach` has zero clients, the message is dropped, and there is no replay when the app reopens. When the user later opens the app, the fired tripwire is still in mw_tripwires localStorage (App.jsx:153-159), the tripwires effect fires (App.jsx:367) and re-syncs it to the backend, re-arming it.
- **Consequence:** One-shot tripwire silently resurrected after every app cold-start that occurred while a push fired in the background — the most common real-world case (phone locked, app not running). User gets repeat alarms for a crossing they were already notified about and believed was consumed.
- **Fix:** Do not rely on a fire-and-forget postMessage for durable one-shot consumption. On app load, reconcile with the server: fetch the authoritative tripwire list from the backend (the rec already had the fired one stripped) and treat the server as source of truth on startup, or have notificationclick / a periodic reconcile write the removal. Alternatively persist a 'consumedMmsi' set the SW writes to (e.g. via IndexedDB/Cache from the SW) that App reads on mount before its first sync.

## 4. [HIGH] Multiple silent catch blocks swallow push/sync failures — user thinks background alerts are armed when they aren't
- **Dimension:** design-ux
- **File:** marine-traffic/src/App.jsx 363, 369; marine-traffic/src/utils/pushSubscribe.js:66-68, 83-85
- **Problem:** syncTripwiresToBackend(...).catch(() => {}) (App.jsx:369) and getExistingSubscription().then(...).catch(() => {}) (App.jsx:363) silently discard errors. If the sync POST fails (offline, 503 'maks-kapasitet', backend down), the UI still shows '🔔 Varsel aktivt 📡' (VesselPanel:254-255) because pushSubActive is true from a prior subscription, even though the new tripwire never reached the server. The user believes background notification is active for this line when the backend has no record of it.
- **Consequence:** Silent failure on a safety-adjacent alerting feature: the user trusts they'll be notified in the background and simply isn't, with no indication anything went wrong.
- **Fix:** Surface sync state per tripwire: track a 'synced | syncing | failed' status and reflect it in the armed badge (e.g. 📡 vs ⚠ 'Ikke synkronisert — trykk for å prøve igjen'). On catch, set a visible error and offer retry. Never show the '📡 backend active' affordance unless the latest sync for THAT tripwire succeeded.

## 5. [MEDIUM] 60s cooldown is dead code for one-shot tripwires and masks legitimate re-fires
- **Dimension:** correctness
- **File:** src/hooks/useTripwireAlerts.js 13, 33-35 (mirror: bg-poll-trigger.mjs:9, 161-163)
- **Problem:** Tripwires are one-shot: handleCross (App.jsx:289-292) deletes the tripwire on the first cross, and the backend filters fired tripwires out of rec.tripwires (bg-poll:202-205). So the per-vessel 60s COOLDOWN_MS in both detectors can never actually suppress a second fire of the SAME wire (the wire is gone). The cooldown comment claims it stops a vessel 'vibrating along the line' from spam-triggering, but since one-shot delete already does that, the cooldown only adds risk: lastFireRef/st.lastFired persist across wire re-arming, so if the user deletes and immediately re-draws a wire for the same MMSI within 60s and the vessel crosses, the frontend cooldown (lastFireRef.current is component-instance state, survives because the hook isn't remounted) can suppress the legitimate new crossing.
- **Consequence:** Re-arming a wire on the same vessel within 60s of a prior crossing can silently swallow the next crossing.
- **Fix:** Remove the cooldown from the in-app hook (one-shot semantics make it redundant), or reset lastFireRef when the tripwire identity/createdAt changes so a freshly re-armed wire is not gated by the prior wire's fire time. Backend st.lastFired[mmsi] should likewise be cleared when a new tripwire for that mmsi arrives.

## 6. [MEDIUM] In-app watcher seeds prevPos from a stale/dead reported position, fabricating a phantom crossing
- **Dimension:** correctness
- **File:** src/hooks/useTripwireAlerts.js 22-38; App.jsx:135-144, 326-336
- **Problem:** prevPosRef is seeded with whatever the first non-null vessel position is, with no freshness check. liveSelectedForTw can be a vessel whose last AIS report is up to DEAD_MS (90 min) old (App.jsx keeps vessels until age>DEAD_MS and only flags stale). Scenario: user selects a vessel, arms a wire; the vessel's marker is frozen at an old reported position (prevPos). A fresh poll then lands a position that legitimately jumped far (because the report was stale) — the straight segment between old and new can cross the wire even though the vessel never actually traversed it near the line. Unlike MapView, the hook does NOT honor the DR resume-freeze or staleness, so it computes crossings on jumpy reported data the rest of the app deliberately freezes.
- **Consequence:** False tripwire alerts (and immediate one-shot deletion of the wire) caused by AIS data gaps/teleport, especially after the phone wakes from sleep.
- **Fix:** Skip detection when vessel.stale is set or when (now - Date.parse(vessel.timestamp)) > STALE_MS; reset prevPosRef when a stale/dead gap is detected so the next fresh pair is used as the new baseline rather than bridging across a data gap.

## 7. [MEDIUM] tripwire-fired SW broadcast deletes the wire even when the crossing was for a different selected vessel state, and races local detection
- **Dimension:** correctness
- **File:** src/sw.js / App.jsx sw.js:52-53; App.jsx:200-203
- **Problem:** On push, the SW posts tripwire-fired to ALL clients, and App.jsx unconditionally calls handleRemoveTripwire(mmsi). This is correct for one-shot, but there is no guard that the locally-stored wire is the SAME wire the backend fired on. If the user removed the old wire and drew a NEW wire for the same MMSI after the backend already queued a push (alarm mode sends 3 bursts 5s apart, bg-poll:171-175), the late-arriving tripwire-fired message deletes the freshly-drawn wire. The new wire silently disappears with no crossing.
- **Consequence:** A newly re-armed tripwire vanishes when a stale push burst for the previous wire arrives.
- **Fix:** Include the tripwire createdAt/id in the push payload (data.createdAt) and in tripwire-fired; in handleRemoveTripwire only delete if the stored wire's createdAt matches the fired one.

## 8. [MEDIUM] Backend marks state changed and persists prev position even for vessels with no tripwire fire, but never records 'first seen' freshness → first poll after arming can immediately fire on a large gap
- **Dimension:** correctness
- **File:** netlify/functions/bg-poll-trigger.mjs 154-163
- **Problem:** Same class of stale-bridge bug as the frontend, on the server. st.lastPositions[mmsi] is set to the current AIS position with no timestamp/age check. BarentsWatch /latest/combined returns the last known position regardless of how old it is. If a vessel's transponder was off and comes back on far away, prev (the pre-blackout position) and cur (post-blackout) form a long segment that can cross the wire though the vessel never crossed it near the line. The backend has no staleness gate at all (unlike MapView's STALE_MS logic).
- **Consequence:** Background false push when a tracked vessel reappears after an AIS blackout.
- **Fix:** Store the AIS msgtime alongside each lastPositions entry and skip crossing evaluation (just refresh the stored position) when either prev or cur report is older than a threshold, or when the time gap between prev and cur exceeds a max (e.g. > a few minutes at the polling cadence).

## 9. [MEDIUM] Backend swallows the wire deletion when push send fails mid-burst, but leaves lastFired set → wire kept but cannot re-fire until cooldown, and partial alarm bursts
- **Dimension:** correctness
- **File:** netlify/functions/bg-poll-trigger.mjs 163, 172-205
- **Problem:** st.lastFired[mmsi] = now is set BEFORE the send loop. If the very first webpush.sendNotification throws a transient (non-404/410) error, sentCount stays 0, firedMmsi does not get the mmsi (so the wire is NOT removed from subs — good, it survives), BUT lastFired was already advanced to now. On the next cron tick the same crossing chord may still intersect (prev not yet updated? actually prev IS updated to cur at line 156, so the chord is gone) — more importantly the user got zero notifications yet the wire remains armed with lastFired set, so a crossing in the next 60s is cooldown-suppressed. The alarm burst can also be partially delivered (1 of 3) then abort, leaving an inconsistent alarm experience.
- **Consequence:** A transient push failure can drop the only notification for a crossing while marking it as fired.
- **Fix:** Only set st.lastFired[mmsi] after at least one successful send; on transient failure leave lastFired untouched so the next tick can retry. Consider not advancing st.lastPositions for the failed mmsi so the crossing chord is preserved for retry.

## 10. [MEDIUM] push-delete-data / push-unsubscribe leave the aggregate state blob entry until next cron tick, and if cron is stopped the data persists indefinitely
- **Dimension:** correctness
- **File:** netlify/functions/push-delete-data.mjs 19-22 (and push-unsubscribe.mjs:20-22)
- **Problem:** Both endpoints delete only the subs blob and rely on bg-poll to prune the orphaned state entry 'next tick'. For GDPR Art 17 (delete-data) this means the user's last positions remain in tripwire-state/all until a cron tick happens to run. If cron is paused/disabled (the band-aid external cron-job.org can be down, and CLAUDE.md notes cron-cost concerns), the deletion never completes — a compliance and correctness gap. There is also no immediate confirmation that state was actually removed despite the response claiming deleted: ['subscription','tripwires','state'].
- **Consequence:** GDPR deletion is incomplete/delayed; user position history can linger in the state blob if cron is not running.
- **Fix:** In push-delete-data, also load tripwire-state/all, delete the endpoint key, and write it back synchronously so deletion is complete on request, rather than asserting it in the response while deferring the work.

## 11. [MEDIUM] Concurrent bg-poll ticks both read 'tripwire-state/all', last writer clobbers the other's lastPositions/lastFired (lost-update on aggregate state)
- **Dimension:** races-state
- **File:** netlify/functions/bg-poll-trigger.mjs 93, 146-211
- **Problem:** The CLAUDE.md/code comment claims 'bg-poll er eneste skriver → ingen race' (line 84-87, 'single writer = no race'). That is FALSE. The guarantee is single-writer-CODE, not single-writer-EXECUTION. The external cron pings every minute and a tick runs ~2s, so normally one finishes before the next. But Netlify has no overlap guard: a slow tick (BW token refresh, slow AIS POST, or — critically — alarm-mode which does `await sleep(5000)` x2 = >10s in the inner loop at lines 174-175) can still be running when the next cron fires. Both invocations execute `stateAll = await stateStore.get('all')` (line 93) reading the SAME snapshot, each mutates its in-memory copy, then each does `await stateStore.setJSON('all', stateAll)` (line 211). Netlify Blobs has NO atomic/conditional writes (stated in prompt). Whichever setJSON lands LAST wins and silently discards the other tick's mutations.
- **Consequence:** Tick A updates lastFired[mmsi]=now after sending a push and removes a one-shot tripwire; Tick B (started from the pre-A snapshot) writes back the old state with lastFired empty. Result: (1) the cooldown record is lost so the SAME crossing re-fires next tick = duplicate/triple alarms; (2) lastPositions regress to a stale snapshot, so a vessel that already advanced past the line gets re-evaluated with an old prev position and can fire AGAIN or miss a crossing entirely; (3) orphan pruning done by one tick is undone by the other. Alarm-mode (>10s runtime) makes overlap near-guaranteed under load.
- **Fix:** Add a real cross-invocation lock since Blobs lacks CAS: write a lease blob (e.g. meta.setJSON('poll-lock',{at:now})) at the very top, read it first and bail if a fresh lock (< ~2-3 min old) exists; clear it in a finally. Better: move the alarm burst OUT of the synchronous state-holding loop (compute all sends, persist state once, then fire bursts), so a tick never holds the read-modify-write window open for 10s. Best per CLAUDE.md: move to an always-on worker on flat-rate host. At minimum, do NOT `await sleep()` between webpush sends while holding the un-persisted stateAll.

## 12. [MEDIUM] Read-modify-write window: webpush + sleep happen BEFORE state is persisted, so a crash/timeout after sending loses lastFired and re-fires
- **Dimension:** races-state
- **File:** netlify/functions/bg-poll-trigger.mjs 163-211
- **Problem:** `st.lastFired[mmsi] = now` (line 163) and the one-shot removal (`firedMmsi.add`, line 194) are only mutations to IN-MEMORY objects. The actual durable writes happen later: `subs.setJSON(key, next)` at line 204 (per-record) and `stateStore.setJSON('all', stateAll)` at line 211 (once, at the very end after the entire records loop). Between marking fired and persisting, the function performs network I/O: `await webpush.sendNotification` and, for alarm mode, `await sleep(5000)` twice. If the function times out (Netlify function wall-clock limit), the AIS fetch for the next record hangs, or the process is killed, the push was ALREADY delivered but neither lastFired nor the tripwire removal is persisted.
- **Consequence:** Push delivered to the user, but on the next tick lastFired is still 0/old and the tripwire is still present → the identical crossing fires again. With alarm mode the burst already sent 1-3 notifications; a re-fire next tick sends another burst. The cooldown (line 162) cannot save you because lastFired was never written. One-shot tripwires are not actually one-shot under any failure between send and final setJSON.
- **Fix:** Persist state incrementally/eagerly: write lastFired and the tripwire removal to durable storage immediately after a successful send, BEFORE the alarm-burst sleeps and before moving to the next record. Or restructure: (a) detect all crossings and update stateAll, (b) persist stateAll + rewrite recs once, (c) only then send pushes (idempotency now protected by persisted cooldown). Sending-then-persisting is the wrong order for at-most-once semantics.

## 13. [MEDIUM] Lost subscription: webpush 404/410 deletes the sub but a concurrent push-subscribe / in-flight sync recreates it; and any non-404 send error aborts the whole record, skipping later tripwires permanently for that tick
- **Dimension:** races-state
- **File:** netlify/functions/bg-poll-trigger.mjs / netlify/functions/push-subscribe.mjs bg-poll:180-193; push-subscribe:82
- **Problem:** Two issues. (1) On 404/410 the tick does `subs.delete(key)` and `delete stateAll[key]` (lines 181-182). If the frontend's periodic sync (App.jsx:367-370) is firing at the same minute, push-subscribe.setJSON (line 82) can recreate the sub blob with the SAME key AFTER bg-poll deleted it — but bg-poll then writes stateAll at line 211 with that key pruned, so the resurrected sub has no state entry (fine) — however the sub was 404/410 (dead endpoint) and is now resurrected as a zombie that will 404 again next tick: delete/recreate flapping wastes work and the user keeps a dead sub. (2) `aborted = true; break` on ANY send error (line 189-193) breaks out of the ENTIRE rec.tripwires loop, not just the current tripwire. A single transient 500/timeout from the push service on tripwire #1 skips evaluation of tripwires #2..#N for that subscriber this tick, AND for tripwire #1 lastFired was already set to now (line 163) before the send failed (line 177), so its crossing is suppressed for 60s despite never delivering.
- **Consequence:** (1) Dead endpoints flap between deleted and recreated, never cleanly pruned, and the user perceives a 'subscribed' state that never delivers. (2) A transient error on one tripwire blinds the poller to all the subscriber's other tripwires for that tick (missed crossings) and burns the cooldown on the failed one (a real crossing during the next 60s is dropped). lastFired is set before confirming delivery — wrong order.
- **Fix:** Set lastFired only AFTER a confirmed successful send (move line 163 to after sentCount>0). On transient (non-404/410) send errors, `continue` to the next tripwire instead of `break`-ing the whole record. For the delete/recreate race, have push-subscribe verify the endpoint isn't being concurrently reaped, or tolerate it by re-validating on send; ideally guard sub deletion + recreation with the same poll lock proposed for the state race.

## 14. [MEDIUM] unsubscribe/delete-data delete the sub but never the state entry; relies on bg-poll prune which may never run for that key if no records remain
- **Dimension:** races-state
- **File:** netlify/functions/push-unsubscribe.mjs / push-delete-data.mjs / bg-poll-trigger.mjs push-unsubscribe:20-23; push-delete-data:19-22; bg-poll:109-116
- **Problem:** Both unsubscribe and delete-data intentionally do NOT touch tripwire-state (documented at unsubscribe lines 20-21, delete-data lines 19-20), delegating cleanup to bg-poll's orphan prune. But bg-poll only prunes inside two paths: the no-records early return (lines 109-116, which prunes against liveKeys) and the end-of-run prune (lines 209-211). The end-of-run prune is only reached if `records.length` was truthy AND mmsis.length truthy AND the AIS/token fetches succeeded (any early `return` at lines 121/125/132 skips pruning). If, after a delete, NO subscriber has any tripwires (records empty) the early-return path DOES prune (good). But if other subscribers exist with tripwires yet the deleted user's state lingers, pruning happens at line 210 — UNLESS a token/AIS error returns early (lines 125/132), in which case state is never written and the orphan persists across every failing tick. For GDPR delete-data this means the user's lastPositions can survive in tripwire-state indefinitely whenever the poller is erroring.
- **Consequence:** GDPR Art 17 violation risk: delete-data returns {deleted:['...','state']} (push-delete-data line 22) asserting state was deleted, but state deletion is deferred to a best-effort prune that can be skipped indefinitely on AIS/token outages. User's last known positions retained after they requested erasure.
- **Fix:** Delete the state entry synchronously in push-delete-data (and unsubscribe): load tripwire-state/all, delete the key, write it back — accept the small extra Blobs op for correctness/compliance. Do not have a GDPR-deletion endpoint claim state was deleted when it only scheduled a maybe-prune. Guard the read-modify-write with the poll lock to avoid clobbering bg-poll.

## 15. [MEDIUM] In-app detection AND backend push both fire — suppressOsNotify only mutes the OS chime, not the duplicate detection
- **Dimension:** design-ux
- **File:** marine-traffic/src/hooks/useTripwireAlerts.js 36-37
- **Problem:** When push is subscribed, BOTH the client watcher and the backend cron detect the same crossing. suppressOsNotify only skips fireSingle (the OS notification); onCross still runs locally, AND the backend independently sends a Web Push for the same crossing. The two detectors run on different position samples (client = whatever the foreground poll saw; backend = its own AIS poll) with independent 60s cooldowns that do not coordinate. So a single crossing can produce: (a) the local in-app banner/blink/alarm-bar from onCross, plus (b) a backend push which the SW turns into ANOTHER in-app banner (tripwire-event) when focused, plus (c) a tripwire-fired message. The user can literally see two banners for one crossing.
- **Consequence:** The user's exact complaint ('alerts pop up in several annoying places'): one vessel crossing fires up to two visible banners plus an OS-level surface, with no dedup between the local watcher and the backend.
- **Fix:** Pick ONE source of truth per app state. When the app is foreground AND push is active, the backend push is the canonical event — have onCross NOT render its own banner/alarm in that case and rely solely on the SW 'tripwire-event' path (or vice-versa). Gate the local onCross visual surfaces on !pushSubActive, the same way the OS notif is gated. Better: when push is active, the client watcher should do nothing at all except optimistic UI removal, and let the SW message drive all foreground UI so there is exactly one rendering path.

## 16. [MEDIUM] Alarm-modus sends a burst of 3 separate pushes 5s apart — each is its own dismissable OS notification
- **Dimension:** design-ux
- **File:** marine-traffic/netlify/functions/bg-poll-trigger.mjs 15-16, 171-192
- **Problem:** Alarm mode is simulated by sending ALARM_BURST_COUNT=3 pushes with ALARM_BURST_GAP_MS=5000 between them. All three share tag `tripwire-${mmsi}` with renotify, so on Android they re-alert the same notification 3 times over 10s; on iOS/other UAs they may stack as 3 entries. The code comment itself admits 'Tap Bekreft stopps visuelt... can not avbryte allerede-køet pushes' — i.e. acknowledging does NOT stop the burst. The user gets buzzed 3 times even after they've seen and confirmed the first one.
- **Consequence:** Repeated un-cancellable buzzing is the most viscerally annoying notification anti-pattern; confirming the alert does nothing, which trains users to disable notifications or alarm mode altogether.
- **Fix:** Send ONE push for alarm mode and express urgency through the payload, not repetition: longer/stronger vibrate pattern + requireInteraction + the Bekreft action (already done in the SW). If a repeating audible alarm is truly wanted, loop a sound client-side in the SW/page on receipt of a single push and stop it on the ack action — that way Bekreft actually silences it. Drop the burst entirely, or make the burst cancellable by checking, before each subsequent send, whether the crossing was acked (store an ack flag the ack action POSTs back).

## 17. [MEDIUM] One-shot deletion makes the tripwire line silently vanish with no explanation
- **Dimension:** design-ux
- **File:** marine-traffic/src/App.jsx 285-293
- **Problem:** handleCross deletes the tripwire from state the instant the vessel crosses (next[mmsi] is removed). The drawn line — the user's primary visual artifact — disappears from the map. The only feedback is a 10s red blink on recentCrossMmsi (TripwireOverlay line 53/65), but the blink targets a line that is being deleted in the same tick, so the line is gone almost immediately. Nothing tells the user 'this was a one-shot alert and it has now fired and been removed.' A user who set a line and looks back later just sees it missing and may assume a bug.
- **Consequence:** The core mental model (I drew a line, it's gone) is broken with zero feedback; the user cannot tell a fired alert from a lost one, undermining trust in the whole feature.
- **Fix:** Don't delete on cross. Instead set armed:false and stamp lastCrossed (see separate finding), and render disarmed lines in a distinct 'spent' style (e.g. solid grey, checkmark, 'Krysset 12:34 — trykk for å re-armere'). Keep them on the map until the user dismisses or re-arms. If auto-removal is desired, first show a persistent toast/banner: 'Linje for <name> utløst kl 12:34 og fjernet' with an Undo. The vanishing must be announced and reversible.

## 18. [MEDIUM] Confusing/duplicate state after resume: backend can re-fire a crossing the client already removed, and vice-versa
- **Dimension:** design-ux
- **File:** marine-traffic/src/sw.js 50-64; marine-traffic/netlify/functions/bg-poll-trigger.mjs:197-205
- **Problem:** One-shot removal is coordinated by best-effort messages, not a transaction. Client handleCross deletes locally and (on next effect) syncs the now-shorter list to the backend; backend independently deletes fired tripwires from its rec and posts tripwire-fired. If the app was backgrounded/closed when the backend fired, the next foreground sync (App.jsx:367-370) re-uploads whatever is still in localStorage. There's a window where the client's localStorage still has the tripwire (handleCross only runs in foreground) so the post-resume sync re-arms it on the backend, which can then fire a SECOND push for a crossing that already happened. The 60s cooldowns are per-detector and don't survive this.
- **Consequence:** After closing and reopening the app, users can get a stale duplicate alert for an event that already fired, reinforcing the 'alerts everywhere' complaint.
- **Fix:** Make crossing idempotent with a server-authoritative event id. Backend should keep a short-lived 'firedAt' per (endpoint,mmsi) and refuse to re-arm an mmsi whose firedAt is within a dedup window even if the client re-syncs it. On the client, when receiving tripwire-fired, persist a 'consumed' marker so a resume-sync doesn't re-upload it. Reconcile from the server response (it already returns the trimmed list) instead of blindly pushing localStorage.

## 19. [MEDIUM] Dead code from continuous→alarm migration: stopAlarm/isAlarmPlaying no-ops and the whole alarm-bar are now meaningless
- **Dimension:** design-ux

## 22. [LOW] Foreground in-app detector and backend both fire for the same crossing → double notification
- **Dimension:** correctness
- **File:** src/hooks/useTripwireAlerts.js 18, 36; App.jsx:335; bg-poll-trigger.mjs:146-205
- **Problem:** suppressOsNotify is wired to pushSubActive (App.jsx:335), so when push is active the in-app hook skips fireSingle but still runs onCross. ## 25. [LOW] prev===cur short-circuit uses exact float equality, so a vessel that crosses and returns between polls (or jitters) can be missed
- **Dimension:** correctness
- **File:** netlify/functions/bg-poll-trigger.mjs 159 (mirror useTripwireAlerts.js:31)
- **Problem:** The detector only ever tests the straight chord between the previous polled position and the current one. At 1-3 min cron cadence (CLAUDE.md says interval was moved to 3 min), a fast vessel can cross the wire and come back, or cross and the next sample lands on the same side, so the chord prev->cur does not intersect the wire even though a real crossing happened. ## 29. [LOW] Cooldown (60s) is redundant and confusing under one-shot semantics
- **Dimension:** design-ux
- **File:** marine-traffic/src/hooks/useTripwireAlerts.js 13, 33-35; marine-traffic/netlify/functions/bg-poll-trigger.mjs:9, 161-163
- **Problem:** Both client and backend keep a 60s per-vessel COOLDOWN against re-fire. But tripwires are now one-shot — after the first crossing the line is deleted (client) / filtered out of rec (backend). So the cooldown can only ever matter in the narrow race where deletion hasn't propagated yet, and even then it's a second, uncoordinated dedup layer on top of the one-shot removal. Two overlapping dedup mechanisms (cooldown + one-shot delete) with independent per-detector state are hard to reason about and were the kind of thing that let duplicate alerts slip through.
- **Consequence:** Redundant, uncoordinated dedup layers are exactly where duplicate-alert bugs hide; simplifying to one mechanism reduces both bugs and reasoning load.
- **Fix:** Decide on one dedup mechanism. If one-shot is the contract, drop the cooldown and instead dedup on a single fired flag/event-id shared between client optimistic-removal and backend authoritative-removal. If you keep a cooldown as a race guard, make it explicit that it's only a race guard and key it to the same event id, not wall-clock per vessel.
